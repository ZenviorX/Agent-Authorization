# Task21：对比实验体系建设与基线方法验证

## 一、任务背景

在前一阶段中，项目已经完成了安全评测体系 V2，能够通过 70 条样例对 Agent 工具调用安全网关进行自动化评测，并生成 CSV、JSON、Markdown 和 HTML 等多格式结果文件。

但是，仅证明本系统在自有评测集上表现良好还不够。对于参赛作品而言，还需要进一步回答一个关键问题：本项目方案相比基础方法到底强在哪里。

因此，本阶段围绕“对比实验体系”进行建设，在同一套评测样例上引入无防护 Agent 和简单关键词规则两类基线方法，并与本项目 Agent Authorization Gateway 进行横向比较，从而更清晰地展示系统在攻击拦截、误放行控制、正常任务可用性和运行时开销方面的优势。

## 二、任务目标

本任务的目标是构建一套可复现、可量化、可展示的对比实验框架。

具体包括：

1. 实现无防护 Agent 基线；
2. 实现简单关键词规则基线；
3. 在同一套 70 条安全评测样例上运行三组方案；
4. 自动统计攻击误放行率、攻击阻断或确认率、正常误拒绝率和平均检测延迟；
5. 生成 CSV、JSON、Markdown 和 HTML 对比实验结果；
6. 将对比实验接入一键复现实验脚本；
7. 补充对比实验说明文档，为作品报告和答辩提供支撑。

## 三、主要工作内容

### 1. 新增无防护 Agent 基线

本阶段新增 allow_all 基线策略，用于模拟没有任何安全防护机制的 Agent 工具调用行为。

该策略不检查用户身份、工具类型、请求参数和风险内容，所有工具调用请求均直接放行。它可以作为最弱安全基线，用来说明如果缺少工具调用前置安全治理，Agent 在面对攻击样例时会产生多高的误放行风险。

### 2. 新增简单关键词规则基线

本阶段新增 keyword 基线策略，用于模拟传统轻量级规则检测方法。

该策略将请求内容拉平成文本，并检查其中是否包含 secret、password、token、api_key、rm -rf、powershell、curl、ignore previous instructions、绕过权限检查等危险关键词。若命中关键词则拒绝，否则直接放行。

该方案可以拦截一部分显式危险请求，但难以处理上下文风险、低置信度 Agent 计划、未知工具调用和更加隐蔽的绕过型攻击。

### 3. 新增对比实验主脚本

本阶段新增 run_baseline_comparison.py 脚本，用于在同一套 gateway_cases*.json 样例上统一运行 allow_all、keyword 和 gateway 三组方案。

脚本会自动读取 security_cases 目录下的所有评测样例，并分别统计每种方案的总体一致率、攻击阻断或确认率、攻击误放行率、正常任务误拒绝率、平均检测延迟、决策分布和分类准确率。

### 4. 新增多格式对比实验结果输出

对比实验运行后，会自动生成 baseline_comparison_results.csv、baseline_comparison_summary.json 和 baseline_comparison_report.md 三类结果文件。

其中，CSV 文件保存每条样例在不同策略下的详细决策结果；JSON 文件保存结构化统计摘要；Markdown 文件以表格形式展示三组方案的总体对比结果，可直接用于作品报告和阶段总结。

### 5. 新增对比实验 HTML 仪表盘

本阶段新增 generate_baseline_comparison_dashboard.py 脚本，用于根据对比实验结果自动生成 baseline_comparison_dashboard.html。

该页面以卡片、表格和柱状条的形式展示无防护 Agent、关键词规则和本项目安全网关在攻击误放行率、攻击阻断或确认率、正常任务误拒绝率和平均检测延迟等指标上的差异。

该仪表盘可以用于 README 展示、作品报告截图和答辩演示，使对比实验结果更加直观。

### 6. 接入一键复现实验脚本

本阶段进一步修改 scripts/run_full_gateway_eval.ps1，将原有安全评测、评测仪表盘生成、对比实验和对比实验仪表盘生成整合到同一个脚本中。

运行该脚本后，系统可以一键完成完整实验流程，生成 gateway_eval_dashboard.html 和 baseline_comparison_dashboard.html 两个可视化页面，进一步提升实验复现效率和工程完整性。

### 7. 补充对比实验说明文档

本阶段新增 docs/baseline_comparison_experiment.md，对对比实验目标、对比方案设计、评测指标、实验结果分析、实验输出和后续扩展方向进行了说明。

该文档可以作为作品报告中“对比实验设计与结果分析”部分的基础材料，也可以在答辩时用于解释为什么本项目方案相较基础方法具有优势。

## 四、实验结果

在当前 70 条统一评测样例上，三组方案的结果如下：

| 方案 | 总样例数 | 总体一致率 | 攻击阻断/确认率 | 攻击误放行率 | 正常误拒绝率 | 平均延迟 |
|---|---:|---:|---:|---:|---:|---:|
| allow_all | 70 | 21.43% | 0.00% | 100.00% | 0.00% | 0.000 ms |
| keyword | 70 | 74.29% | 75.51% | 24.49% | 0.00% | 0.004 ms |
| gateway | 70 | 100.00% | 100.00% | 0.00% | 0.00% | 0.115 ms |

从实验结果可以看出，无防护 Agent 虽然没有正常误拒绝问题，但会直接放行全部攻击样例，攻击误放行率达到 100.00%，无法提供任何有效安全防护。

简单关键词规则能够拦截部分显式危险请求，使攻击误放行率下降到 24.49%，但仍然存在明显漏检问题。该结果说明，仅依赖关键词匹配难以覆盖上下文风险、低置信度计划、未知工具和绕过型攻击。

本项目 Agent Authorization Gateway 在同一套样例上实现了 100.00% 的攻击阻断或确认率，攻击误放行率为 0.00%，正常任务误拒绝率为 0.00%，平均检测延迟为 0.115 ms。该结果说明，本项目方案在当前评测集上能够兼顾安全性、可用性和运行效率。

## 五、任务价值

本阶段工作使项目从“单系统自测”进一步升级为“多方案横向对比验证”。

在参赛展示中，单纯展示本系统通过了若干测试样例，容易被认为是自定义规则下的结果。而引入 allow_all 和 keyword 两类基线之后，可以更清楚地证明本项目方案相较基础方法的提升幅度。

该对比实验可以直接回答以下问题：

1. 为什么不能让 Agent 直接调用工具；
2. 为什么简单关键词规则不够；
3. 本项目安全网关相比基础方法强在哪里；
4. 本项目是否同时兼顾安全性和正常任务可用性；
5. 本项目是否具备低延迟运行时部署的可能性。

因此，本阶段成果不仅增强了项目实验可信度，也为后续作品报告、答辩展示和国赛评审问答提供了重要支撑。

## 六、后续优化方向

后续可以继续扩展更多基线方法和消融实验。

例如，可以增加传统 RBAC 权限控制基线，用于比较单纯身份权限判断与本项目任务上下文风险判断之间的差异。

也可以增加仅任务合约、仅风险评分、仅关键词规则、完整安全网关等消融实验，用于证明本项目中不同模块各自的贡献。

此外，还可以设计多步攻击链对比实验，比较不同方案在连续工具调用过程中的防护能力，例如“读取外部内容、受到提示注入影响、访问敏感文件、外发敏感信息”的完整攻击路径。

通过这些扩展，项目可以进一步形成“评测集、对比实验、消融分析、可视化展示”的完整实验体系，从而更接近国赛一等奖作品所需要的研究深度和工程完整度。
