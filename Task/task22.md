# Task22：Secure Agent Runtime 真实工具运行时接入层建设

## 一、任务背景

在前一阶段中，项目已经完成了安全评测体系和对比实验体系，能够证明 Agent Authorization Gateway 在工具调用安全检查、攻击拦截和基线对比方面具有较好的效果。

但是，如果系统只停留在“对工具调用请求进行风险判断”的层面，仍然更像一个安全检测模块。为了进一步提升项目的系统完整性和参赛竞争力，本阶段开始建设 Secure Agent Runtime，将 Gateway 的安全决策与真实工具执行流程连接起来，使项目从“工具调用安全检测器”升级为“Agent 工具调用运行时安全中间层”。

本任务的核心目标是：让 Agent 产生的工具调用请求在真正执行前必须经过 Gateway 检查，只有 allow 请求才会进入受控工具执行器，confirm 请求进入人工确认等待，deny 请求直接阻断，从而形成完整的“检查、决策、执行或阻断”运行时链路。

## 二、主要工作内容

### 1. 新增运行时模块目录

本阶段新增 backend/runtime 运行时模块，并创建 runtime_workspace 作为受控工具执行工作区。

runtime_workspace 用于隔离真实文件读写操作，避免工具执行直接作用于项目根目录或系统敏感路径，为后续接入真实 Agent 工具提供基础安全边界。

### 2. 新增 SafeToolExecutor 受控工具执行器

本阶段新增 backend/runtime/tool_executor.py，实现 SafeToolExecutor。

该执行器支持 file.read、file.write、shell.run、email.send 和 db.query 等工具类型。其中，文件读写被限制在 runtime_workspace 内，防止路径逃逸；shell.run 采用白名单命令策略，只允许 echo、dir、ls、pwd、whoami 等低风险命令；email.send 和 db.query 采用模拟执行方式，不真实发送邮件或连接数据库。

通过该设计，系统能够在演示和评测环境中进行真实工具执行链路验证，同时避免产生高危副作用。

### 3. 新增 SecureAgentRuntime 运行时编排器

本阶段新增 backend/runtime/secure_agent_runtime.py，实现 SecureAgentRuntime。

该模块负责接收 Agent 工具调用请求，先调用 Gateway 进行安全检查，然后根据 Gateway 决策执行不同流程：

- allow：进入 SafeToolExecutor 执行工具；
- confirm：不执行工具，进入人工确认等待状态；
- deny：直接阻断，不触发工具执行。

该模块将原先单独存在的安全网关能力接入到真实执行链路中，使系统具备运行时治理能力。

### 4. 新增运行时演示脚本

本阶段新增 examples/secure_runtime_demo.py，用于展示 Secure Agent Runtime 的基本运行效果。

演示脚本包含正常读取公开文件、管理员执行低风险命令、尝试读取敏感路径、尝试外发敏感信息等案例，用于说明系统如何根据 Gateway 决策决定是否执行工具调用。

### 5. 新增运行时批量评测脚本

本阶段新增 experiments/run_secure_runtime_eval.py，用于在已有 70 条安全样例上批量验证 Secure Agent Runtime 的运行效果。

该脚本会统计总样例数、实际执行样例数、进入人工确认样例数、被网关阻断样例数、攻击样例执行率、攻击样例未执行率、正常样例执行率、平均运行时耗时和运行时执行状态分布等指标。

## 三、实验结果

当前 Secure Agent Runtime 批量评测结果如下：

- 总样例数：70
- 实际执行样例数：5
- 进入人工确认样例数：34
- 被网关阻断样例数：31
- 总体执行率：7.14%
- 总体确认率：48.57%
- 总体阻断率：44.29%
- 平均运行时耗时：0.188 ms

在攻击样例执行控制方面：

- 攻击样例数：49
- 攻击样例实际执行数：0
- 攻击样例未执行数：49
- 攻击样例执行率：0.00%
- 攻击样例未执行率：100.00%

该结果说明，在 Secure Agent Runtime 中，攻击请求即使进入系统，也不会触发真实工具执行。系统已经不只是对攻击请求进行评分或分类，而是能够在运行时层面阻止危险工具调用落地执行。

在正常样例方面：

- 正常样例数：15
- 正常样例实际执行数：5
- 正常样例未执行数：10
- 正常样例执行率：33.33%

部分正常样例进入 confirm 状态，说明当前 Gateway 策略相对谨慎。该现象可以作为后续优化方向，通过细化正常任务授权合约、补充运行时测试文件和调整风险阈值，进一步提升正常任务执行率。

## 四、任务价值

本阶段工作显著提升了项目的大模块完整性。

在此之前，项目已经能够判断一个工具调用是否安全，但还没有真正形成“安全检查之后是否执行工具”的运行时闭环。本阶段将 Gateway、安全决策和受控工具执行器串联起来，使系统具备了更完整的 Agent 工具调用治理能力。

这一改进使项目可以在答辩中更明确地表达为：

本项目不是简单的规则检测器，而是面向 AI Agent 工具调用的运行时安全中间层。Agent 的每一次工具调用在执行前都必须经过 Gateway 检查，只有安全请求才会进入受控执行器，高风险请求会被人工确认或直接阻断。

该能力对于冲击国奖非常重要，因为它体现了项目从“评测型原型”向“可落地运行时系统”的升级。

## 五、当前不足与后续优化方向

当前 Secure Agent Runtime 仍属于第一版实现，后续可以继续从以下方向增强：

1. 补齐 runtime_workspace 中的正常测试文件，使被允许的正常 file.read 请求全部执行成功；
2. 将 confirm 状态与已有人工确认队列进一步打通，实现确认后再执行工具；
3. 将运行时执行记录写入审计日志和哈希链，形成完整可追踪链路；
4. 增加 MCP 工具适配层，使外部工具调用也能经过 Secure Agent Runtime；
5. 接入真实 LLM Agent，使模型生成的工具调用真正通过 Gateway 再执行；
6. 增加运行时 HTML 仪表盘，展示执行、确认、阻断和攻击未执行率。

通过这些后续改进，项目可以进一步从“安全网关原型”升级为“Agent 工具调用安全运行时框架”，为后续冲击决赛和国奖提供更强支撑。
