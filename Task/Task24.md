# Task24：用户组清理及 Results 仪表盘改进总结

## 一、任务背景

本阶段完成两类整理工作：一是统一项目中的用户组/角色模型，二是升级 CI 评测结果仪表盘。

此前项目中存在 student、teacher、demo_user、root 等实验身份残留，容易导致权限模型解释不清。当前项目不能凭空判断用户现实职业，因此不应按学生、老师等业务身份分配权限，而应采用类似操作系统的 user/admin 权限模型。

## 二、用户组/角色模型清理

1. 权限主体统一收敛为 user 和 admin。
2. active 代码、测试、样例和演示入口中不再把 student、teacher、demo_user、demo_admin、root、alice 作为权限主体。
3. 策略层和测试请求中的 user 字段应只使用 user 或 admin。
4. 明确区分权限主体和业务文本：权限主体只能是 user/admin；邮箱地址、样例 ID、历史记录等不作为权限判断依据。
5. Task 文件夹作为历史开发记录保留，不删除、不重写旧记录。

## 三、Results 仪表盘改进

1. 保留 Results/ 作为 CI 实验结果归档目录。
2. GitHub Actions 每次运行后继续生成 Result_XXX.html 和 Result_XXX.json。
3. HTML 用于人工查看和答辩展示，JSON 用于后续趋势分析和自动化统计。
4. 新增质量门禁，用于判断 pytest、安全样例、误放行率和误拒率是否满足要求。
5. 新增安全评分，将样例通过率、攻击阻断率、风险误放行率、正常误拒率和 pytest 状态汇总为 0 到 100 的分数。
6. 新增安全决策矩阵，用于直观展示攻击样例是否被拦截、正常样例是否被误杀。
7. 仪表盘界面中文化，优化表格排版，扩大判定原因列宽度，支持横向滚动和筛选。

## 四、当前效果

完成后，项目权限模型更加清晰，CI 评测结果展示更加直观。Results 不再只是测试日志，而是更接近面向比赛答辩的安全评测看板。

## 五、后续优化方向

1. 增加历史趋势图，对比多次 Result 的安全评分变化。
2. 将攻击链评测结果纳入同一仪表盘。
3. 给失败样例增加修复建议字段。
4. 将 Results 仪表盘进一步接入前端展示页面。
