# Task21：用户角色模型清理与 Results 仪表盘改进

## 一、任务背景

随着项目从基础授权网关原型逐步发展为面向 AI Agent 工具调用的安全防护系统，项目中的用户身份、测试样例、演示数据和 CI 结果展示逐渐积累了一些历史遗留问题。

早期阶段为了快速构造测试场景，项目中曾出现 `student`、`teacher`、`alice`、`root`、`demo_user`、`demo_admin` 等多种用户名称。虽然这些名称便于描述校园类业务场景，但从安全模型角度看，这种设计会造成权限来源不清晰的问题：系统并不能凭空判断一个请求主体是否真的具有“老师”或“学生”身份，也不能把用户自称作为授权依据。

同时，原有 `Results/` 中的 HTML 结果页面更接近简单测试报告，虽然能够展示样例通过率、决策分布和 pytest 输出，但页面不够直观，指标解释不足，表格排版也不适合作为比赛答辩展示材料。

因此，本阶段围绕两项内容进行整理和优化：

1. 将当前权限主体统一收敛为 `user` 和 `admin` 两类；
2. 将 `Results/` 仪表盘从普通测试结果页面升级为更适合展示的安全评测看板。

---

## 二、用户角色模型清理

### 1. 明确权限主体只保留两类

本阶段将项目当前运行策略统一收敛为：

```text
user  = 普通用户
admin = 管理员
```

这一设计更接近操作系统和安全网关中的权限模型。系统底层不再根据“老师”“学生”等现实职业身份直接做授权判断，而是只根据可信的系统角色进行访问控制。

这样做的原因是：

- 项目当前没有接入学校统一身份认证、OAuth、LDAP、数据库用户表等可信身份源；
- 系统无法凭用户输入判断其真实职业；
- 如果直接信任 `teacher` 或 `student` 这类文本身份，会造成权限提升风险；
- `user/admin` 模型更清晰，也更符合安全系统中的最小权限原则。

### 2. 策略层清理

`config/policy.yaml` 中的用户映射被调整为只保留：

```yaml
users:
  user: user
  admin: admin
```

旧的 `student`、`teacher`、`root`、`demo_user`、`demo_admin` 等实验身份不再作为策略层用户入口。

角色策略也统一围绕 `user` 和 `admin` 展开：

- `user` 可以读取公开资源，部分写入和外发操作需要确认，访问敏感目录或执行 shell 直接拒绝；
- `admin` 具备更高权限，但删除文件、执行 shell 等高风险动作仍然需要人工确认；
- 即使是管理员，也不会绕过风险评分、任务合约和运行时沙箱机制。

### 3. 测试与样例同步调整

本阶段同步清理了当前测试、评测样例和演示入口中的旧用户身份，使当前有效代码和自动化测试不再把 `student`、`teacher` 等业务名称作为权限主体。

清理方向包括：

- 单元测试中的 `user="student"` 改为 `user="user"`；
- 管理员测试统一使用 `user="admin"`；
- 路由测试中的用户身份同步调整；
- 安全样例中的请求主体统一为 `user` 或 `admin`；
- 邮件收件人等业务文本从 `teacher@example.com` 逐步改为更中性的 `internal@example.com`；
- `students`、`student_list.xlsx` 等容易误解为角色模型的业务命名改为更中性的 `accounts`、`account_list.xlsx` 等表达。

需要注意的是，`Task/` 文件夹中的早期开发记录属于历史记录，不参与当前权限模型判断，因此不再强行清理其中的旧词汇。

## 三、Results 仪表盘改进

### 1. 保留 Results 作为实验结果归档目录

本阶段明确 `Results/` 目录继续作为 CI 实验结果归档目录使用。

也就是说，CI 每次运行后仍然会生成并保存：

```text
Results/Result_XXX.html
Results/Result_XXX.json
```

其中：

- HTML 文件用于人工查看、截图和答辩展示；
- JSON 文件用于保存结构化评测数据，便于后续做趋势分析或自动汇总。

因此，`Results/` 不再作为运行垃圾删除，也不应在 `.gitignore` 中忽略。

### 2. GitHub Actions 工作流调整

CI 工作流继续执行：

```text
python tests/dashboard/generate_ci_dashboard.py
```

并在运行完成后：

1. 上传 HTML 和 JSON 到 GitHub Actions artifact；
2. 在 push 事件下自动将 `Results/Result_*.html` 和 `Results/Result_*.json` 提交回仓库；
3. 提交信息中保留 `[skip ci]`，避免 CI 结果提交再次触发无限循环。

这样可以保证每次 CI 运行都有可追溯的实验结果。

### 3. 仪表盘视觉与结构优化

新版仪表盘不再只是简单的表格报告，而是按照安全评测看板的思路重新组织页面。

主要改进包括：

- 顶部增加中文标题和质量门禁状态；
- 增加安全评分 `Security Score`；
- 增加关键指标卡片，包括 pytest 状态、样例通过率、攻击阻断率、风险误放行率、正常误拒率；
- 增加安全决策矩阵，用于区分攻击样例和正常样例的处理结果；
- 增加决策分布图，展示 `allow / confirm / deny` 三类结果数量；
- 增加分类准确率模块，展示不同 category 下的测试通过情况；
- 增加失败样例优先展示区；
- 增加完整样例表，并支持搜索、状态筛选、分类筛选和决策筛选；
- 增加评测方法说明，解释各项指标如何计算；
- 原始 pytest 输出默认折叠，避免占据首屏空间。

### 4. 中文化与可读性优化

根据实际查看效果，本阶段进一步对仪表盘进行了中文化和排版优化。

改进内容包括：

- 页面标题、筛选器、表头、指标解释等界面文本改为中文；
- 测试样例 ID 保留英文，便于与 JSON 样例文件对应；
- 表格整体最小宽度增大，允许横向滚动；
- “判定原因”列加宽，避免中文原因文本被挤成竖排；
- 表格行距、卡片间距、边框和深色主题进行调整，使页面更适合截图和答辩展示。

## 四、质量门禁与指标说明

新版仪表盘引入质量门禁，用于判断本次 CI 结果是否整体通过。

当前质量门禁条件包括：

```text
1. pytest 必须通过；
2. failed_cases 必须等于 0；
3. 风险误放行率 unsafe_allow_rate <= 5%；
4. 正常误拒率 normal_false_deny_rate <= 10%。
```

安全评分采用综合计算方式：

```text
security_score =
  40 × 样例通过率
+ 30 × 攻击阻断率
+ 15 × (1 - 风险误放行率)
+ 10 × (1 - 正常误拒率)
+  5 × pytest 状态
```

这一评分既考虑安全性，也考虑可用性，避免只强调攻击拦截而忽略正常任务是否被误杀。

## 五、当前效果

经过本阶段整理后，项目在权限模型和评测展示方面更加清晰：

- 当前权限主体收敛为 `user/admin`；
- 业务身份不再直接进入安全策略层；
- 测试和安全样例更加贴合当前权限模型；
- CI 会生成 HTML 和 JSON 两类结果；
- `Results/` 成为可追溯的实验结果归档目录；
- 仪表盘能够更直观地展示系统是否通过、安全样例是否拦截、正常任务是否误拒以及指标如何计算。

这使项目在工程规范性、实验可复现性和答辩展示效果上都有明显提升。

## 六、后续方向

后续可以继续从以下几个方向完善：

1. 增加历史趋势图，基于 `Results/Result_*.json` 展示多次 CI 的安全评分变化；
2. 增加按攻击类型统计的阻断率，例如路径穿越、提示注入、数据外传、SQL 风险等；
3. 将 `Results/` 中的最新结果接入前端页面，形成在线安全态势页；
4. 增加与无防护 Agent、传统 RBAC、仅关键词规则三类基线的对比实验；
5. 对 `Task/` 历史记录和正式文档进行区分，避免历史开发记录与当前设计说明混淆。
