本阶段主要围绕项目的“真实运行能力”和“比赛展示效果”进行了完善。之前系统虽然已经具备工具调用授权、风险判断和任务链审计等功能，但在展示层面还偏向接口和模拟逻辑，评委不太容易直观看到工具调用是否真的发生、是否真的经过网关授权。因此这次重点补充了安全沙箱、证据包和展示入口，让项目从单纯的功能演示进一步变成可运行、可复查的系统闭环。

首先对工具执行部分进行了改造，将文件读取、文件写入、文件删除、邮件发送、数据库查询和命令执行统一限制到 runtime_workspace 沙箱目录中。这样做的好处是，系统可以真实地产生文件、邮件 outbox、SQLite 数据库查询结果等运行痕迹，但不会影响电脑上的真实文件和系统环境。比如 email.send 不会真正向外部邮箱发送邮件，而是把邮件内容写入 outbox 目录；db.query 会查询沙箱中的 SQLite 数据库；shell.run 也只允许少量安全命令。这样既增强了真实性，也保证了演示安全。

随后新增了沙箱运行状态相关接口和前端页面。通过 /sandbox-dashboard 页面，可以直接查看 public、private、secret、outbox 和数据库等沙箱目录状态，也能看到文件数量、邮件记录和数据库内容。页面中还加入了“一键生成演示痕迹”功能，点击后可以自动完成公开文件写入、文件读取、沙箱邮件落盘、数据库查询、安全命令执行以及路径穿越拦截测试。这样在答辩时可以直接展示工具调用在沙箱中的真实执行结果，而不是只看后端返回的 JSON。

在沙箱执行基础上，又加入了证据包机制。系统可以将每一次演示过程中的工具名称、调用参数、执行结果、成功或拦截状态保存为 JSON 文件，并计算 SHA256 哈希值。这样一来，工具调用过程不是执行完就结束，而是会留下可复查的证据文件。后续如果需要验证证据包有没有被修改，也可以重新计算哈希值进行校验。这部分主要提升了项目的审计可信度。

之后进一步增加了“Gateway 授权后执行”的证据包。这个功能比普通沙箱证据更重要，因为它记录的是完整的安全链路：工具请求先进入 Gateway，Gateway 根据用户身份、工具类型、路径、外发风险等因素给出 allow、confirm 或 deny 决策，只有 allow 的请求才会进入 ToolExecutor 执行，deny 和 confirm 的请求不会直接执行。通过这个证据包可以证明系统不是绕过授权直接调用工具，而是真正实现了“授权检查—条件执行—证据固化”的闭环。

为了让展示更清楚，还新增了 /authorized-evidence 页面，用来可视化展示每一步工具调用的授权结果和执行状态。页面会显示正常读取公开文件、越权读取敏感文件、路径穿越攻击、提示注入诱导外发、学生执行系统命令等测试场景，并标明每一步是否被 Gateway 放行、是否进入执行器、执行结果是什么。这样可以更直观地说明系统对正常请求和高风险请求的不同处理方式。

最后新增了比赛展示入口页 /showcase，并对自动启动脚本进行了调整。现在运行 start_project.py 后会自动启动后端，初始化沙箱环境，并打开展示入口页和接口文档。/showcase 页面集中放置了授权证据包、沙箱运行态、多步任务链、安全总览和 FastAPI 文档等入口，避免答辩时手动输入多个地址。同时还加入了“自动生成展示报告”功能，可以汇总当前证据包数量、授权证据包数量、工具调用总步数、实际执行步数、阻断步数和哈希校验通过率，并生成 Markdown 和 JSON 两种格式的报告。

总体来看，本阶段主要完成了从“能判断工具调用风险”到“能真实执行、能留下证据、能集中展示”的升级。项目现在不只是能给出 allow、confirm、deny 的判断结果，还能证明每一步工具调用是否经过授权、是否进入沙箱执行、是否被系统拦截，以及整个过程是否可以被后续复查。这对后续写项目书、录制演示视频和答辩展示都有比较直接的帮助。
