# Agent Authorization Gateway 对比实验说明

## 1. 实验目标

本对比实验用于验证 Agent Authorization Gateway 相比基础方案是否具有更强的安全防护能力和更好的综合表现。

在 AI Agent 工具调用场景中，如果缺少安全网关，Agent 可能会直接执行危险命令、读取敏感文件、外发密钥信息或执行越权数据库操作。仅依赖简单关键词规则虽然可以拦截一部分显式风险，但难以处理上下文风险、低置信度计划、绕过型攻击和多因素综合判断。

因此，本实验在同一套 70 条安全评测样例上，对比三种策略的表现：无防护 Agent、简单关键词规则和本项目安全网关。实验重点观察攻击误放行率、攻击阻断或确认率、正常任务误拒绝率和平均检测延迟等指标。

## 2. 对比方案设计

本实验设置三组对比方案。

第一组是 allow_all，即无防护 Agent 基线。该策略不进行任何安全检查，所有工具调用请求都直接放行。它用于模拟缺少安全治理机制时的 Agent 行为，是最弱安全基线。

第二组是 keyword，即简单关键词规则基线。该策略将请求内容拉平成文本，并检查其中是否包含 secret、password、token、rm -rf、powershell、ignore previous instructions、绕过权限检查等危险关键词。若命中关键词则拒绝，否则放行。该方案代表传统的轻量级规则检测方法。

第三组是 gateway，即本项目 Agent Authorization Gateway。该方案综合使用工具授权、风险评分、人工确认、危险行为识别、未知工具失败关闭等机制，对不同类型的工具调用请求进行更细粒度的安全判断。

通过这三组方案，可以形成从“无防护”到“简单规则”再到“综合安全网关”的递进式对比，体现本项目方案相较基础方法的改进价值。

## 3. 评测指标

本实验主要使用以下指标评价不同方案的表现。

总体一致率用于衡量方案的实际决策与预期安全标签的一致程度。该指标越高，说明方案对整体样例的判断越符合预期。

攻击阻断或确认率用于衡量攻击样例中被拒绝或升级人工确认的比例。对于安全网关而言，面对高风险请求时，不一定必须全部直接拒绝，部分请求进入人工确认也是合理的安全处理方式。

攻击误放行率用于衡量攻击样例被直接允许执行的比例。该指标是安全性评价中的关键指标，数值越低越好。如果攻击请求被直接放行，可能导致敏感文件泄露、危险命令执行、数据库破坏或外部信息泄露。

正常任务误拒绝率用于衡量正常样例被直接拒绝的比例。该指标用于评价系统可用性。一个安全方案如果大量拒绝正常请求，即使攻击拦截率较高，也会影响 Agent 的正常工作能力。

平均检测延迟用于衡量方案在单次工具调用安全检查中的时间开销。该指标可以反映方案是否适合作为运行时前置安全网关部署。

## 4. 实验结果分析

在当前 70 条统一评测样例上，allow_all、keyword 和 gateway 三种方案表现出明显差异。

allow_all 方案的总体一致率为 21.43%，攻击误放行率为 100.00%。这说明在没有任何安全防护的情况下，Agent 会直接执行全部工具调用请求，虽然不会误拒绝正常任务，但面对攻击样例时几乎没有任何防护能力。

keyword 方案的总体一致率为 74.29%，攻击阻断或确认率为 75.51%，攻击误放行率为 24.49%。这说明简单关键词规则可以拦截一部分显式风险，例如 password、secret、rm -rf 等明显危险内容，但仍然存在较高的攻击误放行率，难以覆盖低置信度计划、隐蔽型绕过和上下文风险。

gateway 方案的总体一致率为 100.00%，攻击阻断或确认率为 100.00%，攻击误放行率为 0.00%，正常任务误拒绝率为 0.00%，平均检测延迟为 0.115 ms。该结果说明本项目安全网关在当前评测集上能够有效识别高风险工具调用，并在保持较低延迟的同时兼顾正常任务可用性。

从对比结果可以看出，本项目方案相较无防护 Agent 和简单关键词规则，在攻击误放行率、综合判断能力和可解释安全决策方面具有明显优势。

## 5. 实验输出与使用价值

本实验会自动生成 baseline_comparison_results.csv、baseline_comparison_summary.json、baseline_comparison_report.md 和 baseline_comparison_dashboard.html 四类结果文件。

其中，CSV 文件保存每条样例在不同方案下的详细决策结果；JSON 文件保存结构化统计指标；Markdown 文件可直接作为作品报告中的对比实验部分；HTML 仪表盘可用于答辩展示、README 截图和项目成果说明。

该对比实验的价值在于，它不仅展示了本项目系统本身的运行效果，还证明了本项目方案相较基础基线方法的改进幅度。通过同一评测集上的横向对比，可以更清晰地回答“为什么需要本项目安全网关”“相比简单规则强在哪里”等关键问题。

## 6. 后续扩展方向

后续可以继续从以下方向增强对比实验体系。

第一，扩展更多基线方案，例如传统 RBAC 权限控制、仅风险评分不进行人工确认的方案、仅任务合约不进行内容检测的方案等，从而形成更完整的消融实验。

第二，扩展多步攻击链对比实验，比较不同方案在连续工具调用场景下对攻击链的识别能力，例如“读取外部网页内容、受到提示注入影响、访问敏感文件、外发敏感信息”的完整链路。

第三，引入真实大模型生成的工具调用样例，使评测集更加贴近真实 Agent 运行过程，进一步提升实验可信度。

第四，将对比实验结果接入前端页面，使系统不仅能展示单次工具调用的风险决策，也能展示不同安全策略之间的量化对比效果。

通过这些扩展，项目可以进一步从功能型安全系统提升为具有实验验证、对比分析和研究表达能力的完整参赛作品。
