# Agent Authorization Gateway 安全评测体系 V2

## 1. 评测目标

本评测体系面向 AI Agent 工具调用过程中的授权与安全防护问题，重点验证系统在不同风险场景下的前置拦截、风险升级、人工确认和正常任务放行能力。

传统权限控制通常只关注“用户是否有权限”，但 AI Agent 的工具调用具有更强的不确定性。同一个用户在不同任务上下文中可能触发不同风险，同一次工具调用也可能受到提示注入、攻击链诱导、敏感信息外发等因素影响。因此，本项目将评测目标从单一权限判断扩展为“任务上下文、工具能力、输入内容、风险行为和决策结果”的综合评估。

本轮 V2 评测主要回答以下问题：

1. 系统能否识别高风险工具调用；
2. 系统能否拦截或升级确认攻击样例；
3. 系统是否会过度阻断正常任务；
4. 系统是否具备稳定、低延迟的运行时检测能力；
5. 实验结果是否可以自动复现和结构化导出。

## 2. 样例组织方式

评测样例统一存放在 security_cases 目录下，采用 JSON 格式描述。评测脚本会自动读取所有符合 gateway_cases*.json 命名规则的文件。

因此，后续扩展评测集时不需要修改主评测脚本，只需要新增样例文件即可。例如当前项目中已经包含 gateway_cases.json、gateway_cases_v2.json 和 gateway_cases_v3.json。

这种组织方式使评测体系具有较好的可扩展性，能够支持后续继续加入更多攻击类型、真实任务样例和对比实验样例。

## 3. 样例字段设计

每条评测样例主要包含 id、category、description、request 和 expected_decision 等字段。

其中，id 用于唯一标识样例；category 表示样例类别，包括 normal、attack、suspicious 等；description 用于解释样例意图；request 用于模拟 AI Agent 发起的工具调用请求；expected_decision 表示唯一期望决策；expected_decision_in 表示可接受的多个安全决策。

在安全系统中，并非所有高风险请求都必须直接拒绝。有些请求更合理的处理方式是升级到人工确认。因此，评测体系允许 expected_decision_in 同时包含 confirm 和 deny，以更符合真实安全网关的决策逻辑。

## 4. 覆盖场景

当前评测集覆盖 normal、attack、suspicious 三类样例，并进一步细分为正常任务、路径穿越、敏感文件访问、提示注入、敏感信息外发、高危命令执行、SQL 风险、未知工具调用和低置信度 Agent 计划等场景。

正常任务样例包括读取公开文件、写入普通草稿、发送普通内部邮件、执行低风险健康检查命令、普通只读数据库查询等。该类样例用于衡量系统是否存在过度拦截问题。

攻击样例包括路径穿越读取 .env 文件、访问 secret 目录、读取私钥文件、伪造系统指令、诱导忽略安全规则、外发 token/password/api_key、执行 rm -rf 或 curl 管道命令、执行危险 SQL 查询等。该类样例用于验证系统对高风险工具调用的识别和阻断能力。

可疑样例包括缺失关键参数、低置信度 Agent 计划、边界风险请求等。该类样例用于验证系统是否能够在不确定场景下采取更谨慎的确认或拒绝策略。

## 5. 评测指标

当前评测脚本自动计算总体一致率、攻击阻断/升级确认率、攻击误放行率、正常任务误拒绝率和平均检测延迟等指标。

总体一致率表示系统实际决策与预期决策标签的一致程度，用于衡量安全网关在全部样例上的整体判断效果。

攻击阻断/升级确认率表示攻击样例中被系统拒绝或升级人工确认的比例。该指标越高，说明系统对高风险请求的防护能力越强。

攻击误放行率表示攻击样例被系统直接允许执行的比例。该指标越低越好，因为攻击请求一旦被直接放行，可能导致敏感信息泄露、危险命令执行或越权访问。

正常任务误拒绝率表示正常任务被系统直接拒绝的比例。该指标用于衡量系统可用性。安全系统不能只追求高拦截率，也需要避免对正常任务造成过强干扰。

平均检测延迟表示每次工具调用安全检查的平均耗时。该指标用于说明系统是否具备运行时部署的可能性。较低延迟说明安全检查可以作为 Agent 工具调用链路中的前置网关运行。

## 6. 实验输出与复现方式

运行评测脚本后，系统会自动生成 CSV、JSON、Markdown 和 HTML 四类实验结果文件。

其中，gateway_eval_v2_results.csv 保存每条样例的详细检测结果，包括样例编号、来源文件、分类、期望决策、实际决策、是否通过、风险分数、风险等级、检测原因和检测延迟。

gateway_eval_v2_summary.json 保存结构化统计摘要，包括总样例数、通过样例数、总体一致率、攻击阻断/升级确认率、攻击误放行率、正常任务误拒绝率、平均检测延迟和分类准确率等。

gateway_eval_v2_report.md 可以直接作为实验报告片段，用于作品报告中的实验结果说明。

gateway_eval_dashboard.html 是可视化评测仪表盘，可以用于答辩展示、README 截图和项目成果说明。

当前项目还提供了一键复现实验脚本 scripts/run_full_gateway_eval.ps1。运行该脚本后，系统会自动完成评测、生成报告并打开 HTML 仪表盘，降低复现实验的操作成本。

## 7. 当前阶段结果说明

当前评测体系已经完成多源样例自动加载，并对 normal、attack、suspicious 等多类样例进行统一评测。评测集规模已经扩展到 70 条，覆盖正常工具调用、路径穿越、提示注入、敏感信息外发、高危命令执行、SQL 风险、未知工具调用和低置信度 Agent 计划等场景。

在当前测试结果中，系统能够稳定生成 CSV、JSON、Markdown 和 HTML 四类结果文件，并通过一键脚本完成完整复现实验。这说明项目已经从简单功能验证进一步升级为可复现、可扩展、可展示的安全实验框架。

相比只展示少量手动测试样例，本评测体系具有以下优势：样例规模更大，覆盖场景更丰富；评测过程自动化，降低人为选择性展示的影响；输出格式完整，便于报告、答辩和项目文档复用；同时衡量安全性和可用性，避免只追求拦截率而忽视正常任务体验。

## 8. 后续扩展方向

后续可以继续从以下方向增强评测体系：

1. 将评测集扩展到 100 条以上；
2. 增加多步攻击链专用评测集；
3. 增加与传统 RBAC、关键词规则、无防护 Agent 的对比实验；
4. 引入真实大模型生成的工具调用样例；
5. 增加不同风险阈值下的消融实验；
6. 在前端展示评测结果和攻击链时间线。

通过这些扩展，项目可以进一步从“功能型系统”提升为“有实验、有对比、有安全分析的完整研究型作品”。
